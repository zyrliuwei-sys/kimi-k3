# Web & Motion вҖ” и§Ҷйў‘дёҖй”®еӨҚеҲ» (Video-to-Video Replicate)

**Status:** Design вҖ” awaiting user review
**Date:** 2026-07-22
**Owner:** TBD

## 1. Goal

A new authenticated page **`/web-motion`** where a signed-in user uploads a short source video, clicks one button (**еӨҚеҲ»**), and receives a replicated video generated by an AI video-to-video model. Credits are deducted on submit and auto-refunded on failure.

This also **activates the existing but unwired AI scaffolding** (`aiManager` + `ai-tasks` service + Fal provider) so future image/music generation can reuse the same pipeline.

## 2. Locked decisions (from brainstorm)

| Decision         | Value                                                                                                                                                                                               |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Transformation   | Video вҶ’ video AI replicate. User uploads source video only.                                                                                                                                         |
| Extra user input | **None** вҖ” one-click. Backend sends a fixed default prompt.                                                                                                                                         |
| Provider / model | **Fal** (the only wired provider that accepts `video_input`, `src/core/ai/fal.ts:323`). Default model `fal-ai/kling-video/o1/video-to-video/edit` (already referenced in code); admin-configurable. |
| Default prompt   | `"Faithfully recreate this video in high quality, preserving the original content, motion, and composition."`                                                                                       |
| Credits          | Flat per-video cost, admin-configurable (default **10**). Auto-refund on failure (already supported by `ai-tasks` scaffold).                                                                        |
| Access           | Signed-in only (credits required). No freemium trial.                                                                                                                                               |
| Status updates   | **Client polling** (`GET /api/ai-tasks/:id` every ~3s). Webhook deferred.                                                                                                                           |
| Output storage   | Rehost the result video to **R2** via the provider's `saveFiles` injection, return a permanent URL.                                                                                                 |
| Result schema    | Reuse existing `aiTask` columns вҖ” input URL in `options` JSON, output URL in `taskResult` JSON. **No schema change.**                                                                               |

### New factory: `getAIManager(configs, { saveFiles })`

`AIManager` (`src/core/ai/index.ts`) exists but is never populated. Add a factory mirroring `getStorage()` (`src/modules/storage/service.ts:41`):

- Lives in `src/core/ai/index.ts` (keeps core free of module imports; route passes `configs` + `saveFiles`).
- Constructs `FalProvider({ apiKey: configs.fal_api_key, customStorage: !!saveFiles, saveFiles, uuid: getUuid })`, registers it on a fresh `AIManager`, returns the manager. Returns `null` if `fal_api_key` unset.
- `saveFiles` is wired in the **route** (which may import `getStorage` from `@/modules/storage`) as a closure that rehosts each output file to R2 via `storageManager.downloadAndUpload`. Contract: `saveFiles(files: AIFile[]) => Promise<AIFile[] | undefined>` (`src/core/ai/types.ts`). If storage is unconfigured, pass no `saveFiles` вҖ” outputs then use Fal's temporary URLs (degraded but functional).

## 4. Data flow

1. User picks a video вҶ’ `uploadMediaFile` вҶ’ `POST /api/storage/upload-media` вҶ’ R2 вҶ’ `{ type:'video', url }`.
2. User clicks **еӨҚеҲ»** вҶ’ `POST /api/ai-tasks { videoUrl }`.
3. Backend: `createTask` (PENDING + deduct credits) вҶ’ `fal.generate({ mediaType:VIDEO, model, prompt, options:{video_input:[url]} })` вҶ’ store `remoteTaskId` вҶ’ return `{ taskId }`.
4. Client polls `GET /api/ai-tasks/:id` every 3s via TanStack Query `refetchInterval` (disabled once terminal).
5. On `SUCCESS`: `fal.query` returns the output; `saveFiles` rehosts it to R2; `updateTask(SUCCESS, { videoUrl })`; client renders `<video>` + download link.
6. On `FAILED`: `updateTask(FAILED)` revokes the consumed credit; client shows error.

## 6. API contracts

### `POST /api/ai-tasks`

- Auth: required.
- Body: `{ videoUrl: string }` (v1 video only).
- 200 вҶ’ `{ code:0, data:{ taskId, status:'processing' } }`
- 401 / 402 (insufficient credits) / 400 (bad url / Fal not configured) вҶ’ `{ code:-1, message }`

### `GET /api/ai-tasks/:id`

- Auth: required; `task.userId` must equal session user.
- 200 вҶ’ `{ code:0, data:{ status:'processing'|'success'|'failed', videoUrl?, error? } }`

- Errors surfaced via `toast` (insufficient credits вҶ’ link to pricing/billing; provider not configured вҶ’ admin-facing message).

## 8. Error handling & security

- **SSRF guard (critical):** `videoUrl` passed to Fal is fetched server-side by Fal. Restrict accepted `videoUrl` to our own storage origin (configured R2 `publicDomain`, or the local `/uploads` fallback). Reject anything else with 400. This prevents a user from pointing Fal at an internal URL.
- **Fal not configured** (`fal_api_key` unset): `getAIManager` returns null вҶ’ POST returns "AI provider not configured"; page disables еӨҚеҲ».
- **Insufficient credits:** `createTask` throws `'Insufficient credits'` вҶ’ 402-style respErr; client prompts upgrade.
- **Generation failure:** mark `FAILED`, `updateTask` auto-revokes the credit.
- **Polling timeout:** client stops refetching after ~5 min; task continues server-side; user can reopen the page to resume polling (task row persists).
- **Size/duration:** client guards upload size; Fal rejects over-limit inputs вҶ’ surfaced as FAILED/refunded.
- **Authz:** poll route rejects tasks not owned by the caller.

- **Unit (service mapping):** a fake `AIProvider` returning canned `AITaskResult`s; verify status mapping (PENDING/PROCESSING/SUCCESS/FAILED) and that `taskResult.videoUrl` is persisted.
- **Integration (routes, mocked provider):**
  - `POST` creates an `aiTask` row, deducts `costCredits`, returns `taskId`.
  - Insufficient credits вҶ’ no row, no deduction, error.
  - `GET` SUCCESS path: `fal.query` success вҶ’ row SUCCESS, `taskResult.videoUrl` set.
  - `GET` FAILED path вҶ’ row FAILED, credit revoked.
  - Authz: polling another user's task вҶ’ rejected.
- **Manual:** upload a ~5s clip, еӨҚеҲ», verify output plays and downloads; verify credit decremented on success and refunded on a forced failure.
- Verify `pnpm build` passes (project rule #7).
